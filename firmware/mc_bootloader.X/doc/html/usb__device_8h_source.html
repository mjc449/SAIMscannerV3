198  Function:
199  void USBDeviceTasks(void)
200 
201  Summary:
202  This function is the main state machine/transaction handler of the USB
203  device side stack. When the USB stack is operated in "USB_POLLING" mode
204  (usb_config.h user option) the USBDeviceTasks() function should be called
205  periodically to receive and transmit packets through the stack. This
206  function also takes care of control transfers associated with the USB
207  enumeration process, and detecting various USB events (such as suspend).
208  This function should be called at least once every 1.8ms during the USB
209  enumeration process. After the enumeration process is complete (which can
210  be determined when USBGetDeviceState() returns CONFIGURED_STATE), the
211  USBDeviceTasks() handler may be called the faster of: either once
212  every 9.8ms, or as often as needed to make sure that the hardware USTAT
213  FIFO never gets full. A good rule of thumb is to call USBDeviceTasks() at
214  a minimum rate of either the frequency that USBTransferOnePacket() gets
215  called, or, once/1.8ms, whichever is faster. See the inline code comments
216  near the top of usb_device.c for more details about minimum timing
217  requirements when calling USBDeviceTasks().
218 
219  When the USB stack is operated in "USB_INTERRUPT" mode, it is not necessary
220  to call USBDeviceTasks() from the main loop context. In the USB_INTERRUPT
221  mode, the USBDeviceTasks() handler only needs to execute when a USB
222  interrupt occurs, and therefore only needs to be called from the interrupt
223  context.

355  Function:
356  USB_HANDLE USBTransferOnePacket(uint8_t ep, uint8_t dir, uint8_t* data, uint8_t len)
357 
358  Summary:
359  Transfers a single packet (one transaction) of data on the USB bus.
360 
361  Description:
362  The USBTransferOnePacket() function prepares a USB endpoint
363  so that it may send data to the host (an IN transaction), or
364  receive data from the host (an OUT transaction). The
365  USBTransferOnePacket() function can be used both to receive and
366  send data to the host. This function is the primary API function
367  provided by the USB stack firmware for sending or receiving application
368  data over the USB port.
369 
370  The USBTransferOnePacket() is intended for use with all application
371  endpoints. It is not used for sending or receiving application data
372  through endpoint 0 by using control transfers. Separate API
373  functions, such as USBEP0Receive(), USBEP0SendRAMPtr(), and
374  USBEP0SendROMPtr() are provided for this purpose.
375 
376  The USBTransferOnePacket() writes to the Buffer Descriptor Table (BDT)
377  entry associated with an endpoint buffer, and sets the UOWN bit, which
378  prepares the USB hardware to allow the transaction to complete. The
379  application firmware can use the USBHandleBusy() macro to check the
380  status of the transaction, to see if the data has been successfully
381  transmitted yet.
382 
383 
384  Typical Usage
385  <code>
386  //make sure that the we are in the configured state
387  if(USBGetDeviceState() == CONFIGURED_STATE)
388  {
389  //make sure that the last transaction isn't busy by checking the handle
390  if(!USBHandleBusy(USBInHandle))
391  {
392  //Write the new data that we wish to send to the host to the INPacket[] array
393  INPacket[0] = USEFUL_APPLICATION_VALUE1;
394  INPacket[1] = USEFUL_APPLICATION_VALUE2;
395  //INPacket[2] = ... (fill in the rest of the packet data)
396 
397  //Send the data contained in the INPacket[] array through endpoint "EP_NUM"
398  USBInHandle = USBTransferOnePacket(EP_NUM,IN_TO_HOST,(uint8_t*)&INPacket[0],sizeof(INPacket));
399  }
400  }
401  </code>
402 
403  Conditions:
404  Before calling USBTransferOnePacket(), the following should be true.
405  1. The USB stack has already been initialized (USBDeviceInit() was called).
406  2. A transaction is not already pending on the specified endpoint. This
407  is done by checking the previous request using the USBHandleBusy()
408  macro (see the typical usage example).
409  3. The host has already sent a set configuration request and the
410  enumeration process is complete.
411  This can be checked by verifying that the USBGetDeviceState()
412  macro returns "CONFIGURED_STATE", prior to calling
413  USBTransferOnePacket().
414 
415  Input:
416  uint8_t ep - The endpoint number that the data will be transmitted or
417  received on
418  uint8_t dir - The direction of the transfer
419  This value is either OUT_FROM_HOST or IN_TO_HOST
420  uint8_t* data - For IN transactions: pointer to the RAM buffer containing
421  the data to be sent to the host. For OUT transactions: pointer
422  to the RAM buffer that the received data should get written to.
423  uint8_t len - Length of the data needing to be sent (for IN transactions).
424  For OUT transactions, the len parameter should normally be set
425  to the endpoint size specified in the endpoint descriptor.
426 
427  Return Values:
428  USB_HANDLE - handle to the transfer. The handle is a pointer to
429  the BDT entry associated with this transaction. The
430  status of the transaction (ex: if it is complete or still
431  pending) can be checked using the USBHandleBusy() macro
432  and supplying the USB_HANDLE provided by
433  USBTransferOnePacket().
434 
435  Remarks:
436  If calling the USBTransferOnePacket() function from within the USBCBInitEP()
437  callback function, the set configuration is still being processed and the
438  USBDeviceState may not be == CONFIGURED_STATE yet. In this special case,
439  the USBTransferOnePacket() may still be called, but make sure that the
440  endpoint has been enabled and initialized by the USBEnableEndpoint()
441  function first.
442

495 /**************************************************************************
496  Function:
497  void USBDeviceDetach(void)
498 
499  Summary:
500  This function configures the USB module to "soft detach" itself from
501  the USB host.
502 
503  Description:
504  This function configures the USB module to perform a "soft detach"
505  operation, by disabling the D+ (or D-) ~1.5k pull up resistor, which
506  lets the host know the device is present and attached. This will make
507  the host think that the device has been unplugged. This is potentially
508  useful, as it allows the USB device to force the host to re-enumerate
509  the device (on the firmware has re-enabled the USB module/pull up, by
510  calling USBDeviceAttach(), to "soft re-attach" to the host).
511 
512  Precondition:
513  Should only be called when USB_INTERRUPT is defined. See remarks
514  section if USB_POLLING mode option is being used (usb_config.h option).
515 
516  Additionally, this function should only be called from the main() loop
517  context. Do not call this function from within an interrupt handler, as
518  this function may modify global interrupt enable bits and settings.
519 
520  Parameters:
521  None
522 
523  Return Values:
524  None
525 
526  Remarks:
527  If the application firmware calls USBDeviceDetach(), it is strongly
528  recommended that the firmware wait at least >= 80ms before calling
529  USBDeviceAttach(). If the firmware performs a soft detach, and then
530  re-attaches too soon (ex: after a few micro seconds for instance), some
531  hosts may interpret this as an unexpected "glitch" rather than as a
532  physical removal/re-attachment of the USB device. In this case the host
533  may simply ignore the event without re-enumerating the device. To
534  ensure that the host properly detects and processes the device soft
535  detach/re-attach, it is recommended to make sure the device remains
536  detached long enough to mimic a real human controlled USB
537  unplug/re-attach event (ex: after calling USBDeviceDetach(), do not
538  call USBDeviceAttach() for at least 80+ms, preferably longer.
539 
540  Neither the USBDeviceDetach() or USBDeviceAttach() functions are blocking
541  or take long to execute. It is the application firmwares
542  responsibility for adding the 80+ms delay, when using these API
543  functions.
544 
545  Note: The Windows plug and play event handler processing is fairly
546  slow, especially in certain versions of Windows, and for certain USB
547  device classes. It has been observed that some device classes need to
548  provide even more USB detach dwell interval (before calling
549  USBDeviceAttach()), in order to work correctly after re-enumeration.
550  If the USB device is a CDC class device, it is recommended to wait
551  at least 1.5 seconds or longer, before soft re-attaching to the host,
552  to provide the plug and play event handler enough time to finish
553  processing the removal event, before the re-attach occurs.
554 
555  If the application is using the USB_POLLING mode option, then the
556  USBDeviceDetach() and USBDeviceAttach() functions are not available.
557  In this mode, the USB stack relies on the "#define USE_USB_BUS_SENSE_IO"
558  and "#define USB_BUS_SENSE" options in the
559  HardwareProfile [platform name].h file.
560 
561  When using the USB_POLLING mode option, and the
562  "#define USE_USB_BUS_SENSE_IO" definition has been commented out, then
563  the USB stack assumes that it should always enable the USB module at
564  pretty much all times. Basically, anytime the application firmware
565  calls USBDeviceTasks(), the firmware will automatically enable the USB
566  module. This mode would typically be selected if the application was
567  designed to be a purely bus powered device. In this case, the
568  application is powered from the +5V VBUS supply from the USB port, so
569  it is correct and sensible in this type of application to power up and
570  turn on the USB module, at anytime that the microcontroller is
571  powered (which implies the USB cable is attached and the host is also
572  powered).
573 
574  In a self powered application, the USB stack is designed with the
575  intention that the user will enable the "#define USE_USB_BUS_SENSE_IO"
576  option in the HardwareProfile [platform name].h file. When this
577  option is defined, then the USBDeviceTasks() function will automatically
578  check the I/O pin port value of the designated pin (based on the
579  #define USB_BUS_SENSE option in the HardwareProfile [platform name].h
580  file), every time the application calls USBDeviceTasks(). If the
581  USBDeviceTasks() function is executed and finds that the pin defined by
582  the #define USB_BUS_SENSE is in a logic low state, then it will
583  automatically disable the USB module and tri-state the D+ and D- pins.
584  If however the USBDeviceTasks() function is executed and finds the pin
585  defined by the #define USB_BUS_SENSE is in a logic high state, then it
586  will automatically enable the USB module, if it has not already been
587  enabled.
