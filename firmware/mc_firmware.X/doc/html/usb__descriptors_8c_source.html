23 -usb_descriptors.c-
24 -------------------------------------------------------------------
25 Filling in the descriptor values in the usb_descriptors.c file:
26 -------------------------------------------------------------------
27 
28 [Device Descriptors]
29 The device descriptor is defined as a USB_DEVICE_DESCRIPTOR type.
30 This type is defined in usb_ch9.h Each entry into this structure
31 needs to be the correct length for the data type of the entry.
32 
33 [Configuration Descriptors]
34 The configuration descriptor was changed in v2.x from a structure
35 to a uint8_t array. Given that the configuration is now a byte array
36 each byte of multi-byte fields must be listed individually. This
37 means that for fields like the total size of the configuration where
38 the field is a 16-bit value "64,0," is the correct entry for a
39 configuration that is only 64 bytes long and not "64," which is one
40 too few bytes.
41 
42 The configuration attribute must always have the _DEFAULT
43 definition at the minimum. Additional options can be ORed
44 to the _DEFAULT attribute. Available options are _SELF and _RWU.
45 These definitions are defined in the usb_device.h file. The
46 _SELF tells the USB host that this device is self-powered. The
47 _RWU tells the USB host that this device supports Remote Wakeup.
48 
49 [Endpoint Descriptors]
50 Like the configuration descriptor, the endpoint descriptors were
51 changed in v2.x of the stack from a structure to a uint8_t array. As
52 endpoint descriptors also has a field that are multi-byte entities,
53 please be sure to specify both bytes of the field. For example, for
54 the endpoint size an endpoint that is 64 bytes needs to have the size
55 defined as "64,0," instead of "64,"
56 
57 Take the following example:
58  // Endpoint Descriptor //
59  0x07, //the size of this descriptor //
60  USB_DESCRIPTOR_ENDPOINT, //Endpoint Descriptor
61  _EP02_IN, //EndpointAddress
62  _INT, //Attributes
63  0x08,0x00, //size (note: 2 bytes)
64  0x02, //Interval
65 
66 The first two parameters are self-explanatory. They specify the
67 length of this endpoint descriptor (7) and the descriptor type.
68 The next parameter identifies the endpoint, the definitions are
69 defined in usb_device.h and has the following naming
70 convention:
71 _EP<##>_<dir>
72 where ## is the endpoint number and dir is the direction of
73 transfer. The dir has the value of either 'OUT' or 'IN'.
74 The next parameter identifies the type of the endpoint. Available
75 options are _BULK, _INT, _ISO, and _CTRL. The _CTRL is not
76 typically used because the default control transfer endpoint is
77 not defined in the USB descriptors. When _ISO option is used,
78 addition options can be ORed to _ISO. Example:
79 _ISO|_AD|_FE
80 This describes the endpoint as an isochronous pipe with adaptive
81 and feedback attributes. See usb_device.h and the USB
82 specification for details. The next parameter defines the size of
83 the endpoint. The last parameter in the polling interval.
84 
85 -------------------------------------------------------------------
86 Adding a USB String
87 -------------------------------------------------------------------
88 A string descriptor array should have the following format:
89 
90 rom struct{byte bLength;byte bDscType;word string[size];}sdxxx={
91 sizeof(sdxxx),DSC_STR,<text>};
92 
93 The above structure provides a means for the C compiler to
94 calculate the length of string descriptor sdxxx, where xxx is the
95 index number. The first two bytes of the descriptor are descriptor
96 length and type. The rest <text> are string texts which must be
97 in the unicode format. The unicode format is achieved by declaring
98 each character as a word type. The whole text string is declared
99 as a word array with the number of characters equals to <size>.
100 <size> has to be manually counted and entered into the array
101 declaration. Let's study this through an example:
102 if the string is "USB" , then the string descriptor should be:
103 (Using index 02)
104 rom struct{byte bLength;byte bDscType;word string[3];}sd002={
105 sizeof(sd002),DSC_STR,'U','S','B'};
106 
107 A USB project may have multiple strings and the firmware supports
108 the management of multiple strings through a look-up table.
109 The look-up table is defined as:
110 rom const unsigned char *rom USB_SD_Ptr[]={&sd000,&sd001,&sd002};
111 
112 The above declaration has 3 strings, sd000, sd001, and sd002.
113 Strings can be removed or added. sd000 is a specialized string
114 descriptor. It defines the language code, usually this is
115 US English (0x0409). The index of the string must match the index
116 position of the USB_SD_Ptr array, &sd000 must be in position
117 USB_SD_Ptr[0], &sd001 must be in position USB_SD_Ptr[1] and so on.
118 The look-up table USB_SD_Ptr is used by the get string handler
119 function.
120 
121 -------------------------------------------------------------------
122 
123 The look-up table scheme also applies to the configuration
124 descriptor. A USB device may have multiple configuration
125 descriptors, i.e. CFG01, CFG02, etc. To add a configuration
126 descriptor, user must implement a structure similar to CFG01.
127 The next step is to add the configuration descriptor name, i.e.
128 cfg01, cfg02,.., to the look-up table USB_CD_Ptr. USB_CD_Ptr[0]
129 is a dummy place holder since configuration 0 is the un-configured
130 state according to the definition in the USB specification.
