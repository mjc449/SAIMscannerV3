561 /********************************************************************
562 Function:
563  bool USBRemoteWakeupAssertBlocking(void)
564 
565 Summary:
566  Checks if it is currently legal to send remote wakeup signaling to the
567  host, and if so, it sends it. This is a blocking function that takes ~5-6ms
568  to execute.
569 
570 PreCondition:
571 
572 Parameters:
573  None
574 
575 Return Values:
576  true - if it was legal to send remote wakeup signaling and the signaling was sent
577  false - if it was not legal to send remote wakeup signaling (in this case, no signaling gets sent)
578 
579 Remarks:
580  To successfully use remote wakeup in an application, the device must first
581  let the host know that it supports remote wakeup (by setting the appropriate
582  bit in the attributes field of the USB configuration descriptor). Additionally,
583  the end user must allow/enable remote wakeup for the device. Hosts are not
584  obligated to always allow remote wakeup, and the end user can typically enable/disable this
585  feature for each device at their discretion. Under Windows, devices supporting
586  remote wakeup will normally have an extra check-box setting under the device
587  manager properties pages for the device, which the user can change.
588 
589  Additionally, remote wakeup capability requires driver support, in the USB
590  class drivers being used for the device on the host. Under Windows 7, the CDC
591  drivers do not support remote wakeup capability, while other drivers like HID
592  and WinUSB fully support this capability. If the intended application is
593  CDC or based on some other driver not supporting remote wakeup, it is potentially
594  possible to make the device a composite device (ex: CDC+HID) and then use the
595  HID driver as a means of achieving the remote wakeup effect of the host.
596 
597  Sometime prior to entering USB suspend (usually just prior), if the host and
598  driver(s) support remote wakeup, and the user has enabled the feature for the
599  device, then the host will send notification to the device firmware letting it
600  know that remote wakeup is legal. The USBGetRemoteWakeupStatus() API function
601  can be used to check if the host has allowed remote wakeup or not. The application
602  firmware should not send remote wakeup signaling to the host, unless both
603  USBGetRemoteWakeupStatus() and USBIsBusSuspended() return true. Attempting to
604  send the remote wakeup signaling to the host when it is not legal to do so
605  will normally not wake up the host, and will instead create a violation of
606  the USB compliance criteria. Additionally, devices should not send remote
607  wakeup signaling immediately upon entry into the suspended bus state. Based on
608  the USB specifications, the bus must be continuously idle for at least 5ms before
